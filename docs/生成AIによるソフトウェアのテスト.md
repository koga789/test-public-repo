# 生成AIによるソフトウェアのテスト

## テストに関する用語の整理

**テスト対象**：
- テストを実施する範囲や特定のシステム・機能を指す。

**テストアイテム**：
- テスト対象を構成する個々の要素や機能であり、テストで検証すべき具体的な対象のこと。
通常、1つのテスト対象に対し、多数のテストアイテムがある。

**テストケース**：
- テスト対象（関数など）の特定の機能や挙動を確認する1シナリオ。
1つのテストケースには、対象機能を正しく検証するための「テストデータ」「テスト手順」「期待結果」などが含まれる。
- 例）テストケース、（モック対象）、テストデータ、テスト手順、期待結果

**テストデータ**：
- テスト実行時にテスト対象に入力する具体的な値、ファイルなどのデータを指す。

**テスト手順**：
- テストの具体的な実行手順や操作方法。
例えば、「APIにパラメータを付与してリクエストを送信する」など、テスト実施時の流れを具体的に記載する。

**期待結果（期待値）**：
- 正しい動作として想定される出力や挙動を指す。
実際の実行結果と期待結果を比較し、一致すればテストは「合格」と判定できる。

**テストオラクル**：
- 実行結果の正否を判断するための基準や仕組みを指す。
期待結果そのものを指す場合もあれば、アサーションなどの期待結果との比較方法を指す場合もある。

**テストスイート**：
- 1つのテスト対象やテストアイテムを検証するために、複数のテストケースを組み合わせた集合。

**C0/C1とは**：
- C0（命令網羅 / Statement Coverage）：
プログラム内のすべての命令（ステートメント）を少なくとも1回は実行するというテストの網羅基準です。
- C1（分岐網羅 / Branch Coverage）：
プログラム内のすべての分岐（if文などの条件分岐における「真（True）」と「偽（False）」の両方）を少なくとも1回は実行するというテストの網羅基準です。
- C0やC1はテストの手法（ホワイトボックステスト）そのものを指す言葉ではなく、
ホワイトボックステストを実施してプログラムの内部構造を検証する際の**「網羅率（カバレッジ）の指標・基準」**を表す用語です。
- UTは、「ユニットテスト（工程軸）をホワイトボックステスト（視点軸）的に実行する」などと言える。
- つまり、内部のロジックや制御構造を視点とする「ホワイトボックステスト」を行う上で、「どこまでテストすれば十分と言えるか」を測るための基準として「C0（すべての命令を通ったか）」「C1（すべての分岐の分岐先を通ったか）」が存在する、という関係性になります。

## テスト工程の活用シーンにおける生成AIの入出力

**テスト仕様書の生成**：
- 入力：要求定義／設計工程の文書
- 出力：テスト仕様書

**テストコードの生成**：
- 入力：プロダクションコード（Mコード）、API概要、テスト仕様書
- 出力：テストコード

**生成AI利用に際して**：
- 生成AIにテスト対象の分析や考察をさせるプロセス（planモード活用）を経ることで、より網羅的で質の高いテストケースの生成が期待できる。

## テスト仕様書の生成

具体的な利用手順：
1. 入力となるテストベース（ドキュメント、ソースコード）の準備
2. テストベースから仕様把握・目的・テストアイテムの明確化
3. テスト観点・テスト条件の抽出
4. テストケース（概要）の作成
5. テストケース（詳細）の定義
6. テスト仕様書の最終レビューと整合性確認

### Step 1. 入力となるテストベースの準備

テストベースの例：
- 基本設計書(HLD)／詳細設計書(LLD)
- ソースコード

### Step 2. テストベースから仕様把握・目的・テストアイテムの明確化

まず、ドキュメントからテスト対象を特定します：
```markdown
- 仕様を要約しテスト範囲・目的を明確化
- テストアイテムを抽出し、ITEM-XXX形式でIDを付与
- 不明点は「不明」と記録し、後日確認とする
- 出力形式：
  - テストアイテム一覧（ITEM-XXX、名称/範囲）
  - テスト目的・完了基準（箇条書き可）
- 出力は後工程でのID管理・差分チェックに使用
- 入力：テストベース
```

**重要**：不明点は「不明」と記録し、後日確認とすることが重要です。これにより、テストケースの作成や実行時に必要な情報が不足している場合に、適切なタイミングで確認・補足が行えるようになります。

### Step 3. テスト観点・テスト条件の抽出

システムをさまざまな角度（正常/異常/境界値/セキュリティ/性能など）から検討し、どこをどうテストすべきかを網羅的に洗い出します：
```markdown
- ITEM を参照し、正常/異常/境界値/非機能等のテスト観点を洗い出して連番ID（C-XXX）を付与
- 全てのテスト観点がITEMとひも付いていることを確認
- 観点が多い場合はITEM単位で分割する
- 出力形式：テスト観点ID | ITEM-ID | 観点概要
  例）C-001 | ITEM-001 | 正常系：入力値が有効な場合の動作確認
- 差分チェック：ITEMや観点の追加・変更があった場合、IDを基に差分を確認し、テストケースへの影響を評価する
- 入力：Step2出力（テストアイテム一覧）、テストベース
```

**重要**：Step 2で抽出したテストアイテム（ITEM-XXX）と、Step 3で抽出したテスト観点（C-XXX）が明確にひも付いていることを確認することが重要です。

### Step 4. テストケース（概要）の作成

テスト観点・テスト条件をもとに、より具体的なテストケース（「何を確認するか」の単位）としてリストアップします：
```markdown
- 観点ID（C-XXX）を参照し、テストケース（概要）に整理し、連番ID（G-XXX）を付与
- 各ケースがカバーする観点ID（C-XXX）とITEM-IDを明示
- 観点が多い場合は、観点単位で分割する
- 出力形式：ケース概要ID | 観点ID | ITEM-ID | テストケース概要 | 優先度
  例）TC-001 | C-001 | ITEM-001 | 正常系：入力値が有効な場合の動作確認 | 高
- 差分チェック：観点やITEMの追加・変更があった場合、IDを基に差分を確認し、テストケースへの影響を評価する
- 入力：Step3出力（テスト観点一覧）
```

### Step 5. テストケース（詳細）の定義

テストケース（概要）をもとに、実際に実行可能な形に落とし込み、入力データや手順、期待結果などの詳細を明確にします：
```markdown
- テストケース概要ID（G-XXX）を入力データ、前提条件、手順、入力、期待結果、優先度を含む詳細なテストケース（D-XXX）として具体化
- D-XXXは連番で付与し、対応するG-XXXを明示（1つのG-XXXに対して複数のD-XXXが存在する場合あり）
- 出力形式：ケース詳細ID | ケース概要ID | 前提条件 | 入力データ | テスト手順 | 期待結果 | 優先度
  例）D-001 | G-001 | ITEM-001 | 入力値が有効な場合の動作確認 | 高
- 差分チェック：観点やITEMの追加・変更があった場合、IDを基に差分を確認し、テストケースへの影響を評価する
- 入力：Step4出力（テストケース概要）、詳細設計書やソースコードなどのテストベース
```

**重要**：ここで出力される入力データは簡易的なものとなるため、テストデータが大きくなる場合は、別途テストデータ管理表やJSON構造などで管理すること。

### Step 6. テスト仕様書の最終レビューと整合性確認

これまでに作成したテストアイテム、テスト観点、テストケース（概要・詳細）をテスト仕様書としてまとめます：
```markdown
- Step2～Step5の成果物を「テスト仕様書」として章立てでまとめる
- 構成例：
  - 1. はじめに
  - 2. テストアイテム一覧
  - 3. テスト観点一覧
  - 4. テストケース（概要）
  - 5. テストケース（詳細）
  - 6. トレーサビリティ（ITEM → C → G → D）
  - 7. 変更履歴
- 全てのテストケースが最終的にテストケース（詳細）まで落とし込まれていることを確認
- 指摘があれば、Step2～Step5に戻って修正し、整合性を保つ
- 差分チェック：テストアイテムや観点の追加・変更があった場合、IDを基に差分を確認し、テストケースへの影響を評価する
- 出力形式：テスト仕様書（Markdown形式）、表はMermaid形式で出力
- 入力：Step2～Step5の出力（テストアイテム一覧、テスト観点一覧、テストケース概要、テストケース詳細）、レビュー指摘
```

## テストコードの生成

以下の段階的なアプローチでテストコードを生成します：
1. コードの理解と要約
2. テストケースの網羅的な列挙
3. テストコードの生成
4. テストコードの実行結果に基づく改善
5. カバレッジ測定結果に基づく改善

### Step 1. コードの理解と要約

テストコード生成の際の前提情報（コード概要・入出力・振る舞い）を整理します：
````markdown
[背景]
与えられたPythonコードの内容を理解し、次のステップで網羅的なテストケースをつくるために必要な情報を整理します。

[目的]
対象コードの目的、動作、特に重要な処理内容やエッジケースを正確に把握すること。

[指示]
- 与えられたコードを解析し、関数やクラスが何を目的としているのかを明確に記述してください
- 特に条件分岐、ループ、例外処理などの挙動に着目して要約してください
- 入力と出力の関係を明確に説明してください
- 不明確な点や推測が含まれる場合は、その旨を明記してください

[出力形式]
```
- 役割・処理内容：<簡潔な説明>
- 入出力：
  - 入力：<入力の説明>
  - 出力：<出力の説明>
- 特殊な挙動・エッジケース：<説明>
- 推測箇所（あれば）：<説明>
```

[ユーザ入力]
```python
# ここにテスト対象のPythonコードを入力してください
```
````

### Step 2. テストケースの網羅的な列挙

テストケースを網羅的に列挙するための指示例：
````markdown
[背景]
あなたは対象コードの意図を理解し、pytestでユニットテストを作成するためのテストケースを検討しています。

[目的]
テスト仕様書の情報も参照しながら、対象コードに対してテストすべきケースを網羅的に洗い出す。

[指示]
- 正常系、異常系、境界値、例外処理など、さまざまな観点からテストケースを考えてください
- 提供されたテスト仕様書の情報がある場合は必ず参考にし、該当ケースに明示してください
- テスト仕様書に記載のないケースで必要と思われるものも提案してください

[出力形式]
```
- 正常系：
  - ケース1：<説明>
  - ケース2：<説明>
- 異常系：
  - ケース1：<説明>
  - ケース2：<説明>
- 境界値：
  - ケース1：<説明>
  - ケース2：<説明>
- 例外処理：
  - ケース1：<説明>
  - ケース2：<説明>
```

[ユーザー入力]
- 前のステップでコードの要約を出力している場合は、その内容をここに入力してください
- テスト仕様書の関連部分もここに入力してください
````

### Step 3. テストコードの生成

網羅的に列挙したテストケースを、pytestを用いた実行可能なテストコードに変換します：
````markdown
[背景]
これまでに整理したテストケースの情報を元に、pytestを用いて網羅的なユニットテストコードを生成します。

[目的]
pytestで実行可能な網羅性の高いユニットテストコードを生成する。

[指示]
- 前ステップのテストケースをすべてカバーすること
- 同種のテストは`pytest.mark.parametrize`を用いてまとめること
- 明確なアサーションを設定し、各テストケースが期待する挙動を明記すること
- 例外テストは`pytest.raises`を用いて適切に記述すること
- テスト関数名は、テストケースの内容がわかるように命名すること

[出力形式]
```python
# ここにpytestコード
```

[ユーザー入力]
- ステップ2で列挙したテストケースの内容をここに入力してください
- 対象コードのシグネチャ
````

### Step 4. テストコードの実行結果に基づく改善
生成したテストコードを実行し、失敗したテストケースの内容を分析して、テストコードやテストケースの内容を改善します。

`````markdown
[背景]
生成したテストコードを実行した結果、エラーや失敗が発生しました。これらの失敗を分析し、テストコードやテストケースの内容を改善する必要があります。

[目的]
発生した問題を修正し、テストコードを改善する

[指示]
- 提供されたエラーや失敗したテストのログを参考にし、原因を特定してください
- 必要に応じてテストコードの修正や、テストケースの内容の見直しを行ってください
- 修正の際、原因と修正内容をコメントで簡潔に説明してください

[出力形式]
```python
# 修正したテストコード
```

[ユーザー入力]
- テストコードの実行結果のログ
- 修正前のテストコード
`````

### Step 5. カバレッジ測定結果に基づく改善
テストコードのカバレッジを測定し、カバレッジが低い箇所を特定して、テストコードやテストケースの内容を改善します。
````markdown
[背景]
テスト実行後、カバレッジ不足の領域が特定されました。

[目的]
不足したカバレッジを改善するためのテストコードの修正や追加を行う。

[指示]
以下の手順で作業を進めてください：
- カバレッジ測定結果を確認し、カバレッジが低い原因を分析してください
- 必要に応じてテストコードの修正や、テストケースの内容の見直しを行ってください
- 見直しの際、カバレッジが不足している箇所を明確にし、どのようなテストケースを追加・修正したかを説明してください
- 追加・修正したテストケースを実行可能なpytestコードとして出力してください

[出力形式]
カバレッジが不足している箇所の説明：
- <関数名/分岐条件など>：<説明>
追加・修正したテストケースの説明：
- ケース1：<入力値と期待結果の説明>
- ケース2：<入力値と期待結果の説明>
```python
# 追加・修正したテストコード
```

[ユーザー入力]
- カバレッジ測定結果のログ
- 修正前のテストコード
````
