# 生成AIによるソフトウェアのテスト

学習メモです。

テスト対象：
テストを実施する範囲や特定のシステム・機能を指す。

テストアイテム：
テスト対象を構成する個々の要素や機能であり、テストで検証すべき具体的な対象のこと。
通常、1つのテスト対象に対し、多数のテストアイテムがある。

テストケース：
テスト対象（関数など）の特定の機能や挙動を確認する1シナリオ。
1つのテストケースには、対象機能を正しく検証するための「テストデータ」「テスト手順」「期待結果」などが含まれる。
実際の例）テストケース、（モック対象）、テストデータ、テスト手順、期待結果

テストデータ：
テスト実行時にテスト対象に入力する具体的な値、ファイルなどのデータを指す。

テスト手順：
テストの具体的な実行手順や操作方法。
例えば、「APIにパラメータを付与してリクエストを送信する」など、テスト実施時の流れを具体的に記載する。

期待結果（期待値）：
正しい動作として想定される出力や挙動を指す。
実際の実行結果と期待結果を比較し、一致すればテストは「合格」と判定できる。

テストオラクル：
実行結果の正否を判断するための基準や仕組みを指す。
期待結果そのものを指す場合もあれば、アサーションなどの期待結果との比較方法を指す場合もある。

テストスイート：
1つのテスト対象やテストアイテムを検証するために、複数のテストケースを組み合わせた集合。

C0やC1とは：
C0やC1はテストの手法（ホワイトボックステスト）そのものを指す言葉ではなく、
ホワイトボックステストを実施してプログラムの内部構造を検証する際の**「網羅率（カバレッジ）の指標・基準」**を表す用語です。
UTは、「ユニットテスト（工程軸）をホワイトボックステスト（視点軸）的に実行する」などと言える。
つまり、内部のロジックや制御構造を視点とする「ホワイトボックステスト」を行う上で、「どこまでテストすれば十分と言えるか」を測るための基準として「C0（すべての命令を通ったか）」「C1（すべての分岐の分岐先を通ったか）」が存在する、という関係性になります。

テスト仕様書の生成：
・入力：要求定義／設計工程の文書
・出力：テスト仕様書

テストコードの生成：
・入力：プロダクションコード（Mコード）、API概要、テスト仕様書
・出力：テストコード

生成AI利用に際して：
生成AIにテスト対象の分析や考察をさせるプロセス（planモード活用）を経ることで、より網羅的で質の高いテストケースの生成が期待できる。

具体的な利用手順：
①入力となるテストベース（ドキュメント、ソースコード）の準備
②テストベースから仕様把握・目的・テストアイテムの明確化
③テスト観点・テスト条件の抽出
④テストケース（概要）の作成
⑤テストケース（詳細）の定義
⑥テスト仕様書の最終レビューと整合性確認

①入力となるテストベースの準備：
・要求仕様書
・基本設計書／詳細設計書
・ソースコード
・過去の障害票
